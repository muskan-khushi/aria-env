# ARIA Action Space Guide

This document provides a detailed breakdown of the action space within the ARIA compliance environment. The environment expects actions to be submitted as JSON objects conforming to the `ARIAAction` schema.

## Action Structure

Every action requires an `action_type` field. Depending on the `action_type`, additional fields must be provided.

```json
{
  "action_type": "<ActionType>",
  "...": "..."
}
```

## Available Actions

### 1. `request_section`
Instructs the environment to make a specific section of a document visible to the agent. This is how the agent "reads" the corpus.

- **Required Fields:**
  - `document_id` (str): Identifier of the document (e.g., `"privacy_policy"`).
  - `section_id` (str): Identifier of the section (e.g., `"s1"`).
- **Step Reward:** `0.00`
- **When to use:** During the initial **reading** phase to gather context before identifying gaps.

### 2. `identify_gap`
Flags a specific clause in a document as non-compliant or missing a required regulatory component.

- **Required Fields:**
  - `clause_ref` (str): Reference to the specific clause (e.g., `"privacy_policy.s3"`).
  - `gap_type` (Enum): The nature of the gap (e.g., `"data_retention"`, `"consent_mechanism"`).
  - `severity` (Enum): `"high"`, `"medium"`, or `"low"`.
  - `description` (str): A brief explanation of why this clause constitutes a gap.
- **Step Reward:** `+0.20` for exact match, `+0.12` for partial match. False positives incur a `-0.10` penalty.
- **When to use:** During the **auditing** phase to build a list of active findings.

### 3. `cite_evidence`
Validates a previously identified gap by linking it to the exact text in the document corpus. **Identifying a gap earns no final score without a valid citation.**

- **Required Fields:**
  - `finding_id` (str): The ID of the finding generated by `identify_gap`.
  - `passage_text` (str): The exact quotation from the document.
  - `passage_location` (str): The location of the passage (e.g., `"privacy_policy.s3"`).
- **Step Reward:** Up to `+0.12` based on fuzzy matching score (Levenshtein ratio ≥ 0.55).
- **When to use:** Immediately after identifying a gap or during the **audit** phase.

### 4. `submit_remediation`
Provides a proposed amendment or addition to fix an identified gap.

- **Required Fields:**
  - `finding_id` (str): The ID of the finding being remediated.
  - `remediation_text` (str): The proposed legal text or actionable advice.
  - `target_framework` (Enum, optional): The framework guiding the remediation (e.g., `"GDPR"`).
- **Step Reward:** `+0.15` if the text achieves ≥ 70% coverage of canonical keywords for that gap type.
- **When to use:** During the **remediating** phase after gaps have been identified and cited.

### 5. `flag_false_positive`
Retracts a finding that was previously identified, typically if the agent later reads an exception clause that makes the initial finding invalid.

- **Required Fields:**
  - `retract_finding_id` (str): The finding ID to retract.
- **Step Reward:** `+0.05` for a correct retraction (retracting a true finding costs `-0.08`).
- **When to use:** If adversarial red herrings were mistakenly flagged.

### 6. `escalate_conflict`
Flags a contradiction between two regulatory frameworks in scope (e.g., satisfying HIPAA violates GDPR).

- **Required Fields:**
  - `framework_a` (Enum): The first conflicting framework (e.g., `"HIPAA"`).
  - `framework_b` (Enum): The second conflicting framework (e.g., `"GDPR"`).
  - `conflict_desc` (str): Explanation of the contradiction.
- **Step Reward:** `+0.18`
- **When to use:** Generally during the finalization phase of the Hard task.

### 7. `respond_to_incident`
Responds to a live data breach or compliance incident triggered mid-audit (Expert Tier only).

- **Required Fields:**
  - `incident_id` (str): The ID of the active incident.
  - `response_type` (Enum): Type of response (e.g., `"contain_breach"`, `"notify_supervisory_authority"`).
  - `response_detail` (str): Description of the action taken.
- **Step Reward:** `+0.20` for a correct response within the deadline.
- **When to use:** Immediately when an `active_incident` appears in the observation space. Takes priority over all other actions.

### 8. `request_clarification`
- **When to use:** Currently not scored by the baseline grader. Can be used for human-in-the-loop fallback.

### 9. `submit_final_report`
Terminates the episode and triggers the deterministic Grader.
- **Required Fields:** None.
- **When to use:** When the audit is complete or step budget is nearly exhausted.

## Valid Enums

### Gap Types
- `data_retention`
- `consent_mechanism`
- `breach_notification`
- `data_subject_rights`
- `cross_border_transfer`
- `data_minimization`
- `purpose_limitation`
- `dpo_requirement`
- `phi_safeguard`
- `baa_requirement`
- `opt_out_mechanism`
- `audit_log_requirement`
- `availability_control`

### Incident Response Types
- `notify_supervisory_authority`
- `notify_data_subjects`
- `contain_breach`
- `document_incident`
- `engage_dpo`

### Frameworks
- `GDPR`
- `HIPAA`
- `CCPA`
- `SOC2`

### Severities
- `high`
- `medium`
- `low`
